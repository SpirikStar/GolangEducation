## 1. Введение

**Алгоритм** — конечная последовательность однозначно определённых шагов, преобразующая входные данные в выходные за конечное время.

**Проектирование алгоритма** — это не «сразу писать код», а цепочка решений:

| Этап | Вопрос |
|------|--------|
| Постановка | Что на входе и выходе? Какие ограничения? |
| Модель | Какие операции допустимы и сколько они стоят? |
| Идея | Какой подход: жадный, разделяй-и-властвуй, DP, граф? |
| Корректность | Почему результат всегда верный? |
| Сложность | Сколько времени и памяти? |
| Код | Как выразить это идиоматично на Go? |

Go хорошо подходит для алгоритмических задач: статическая типизация, сборщик мусора, встроенные слайсы и `map`, пакет `testing`, профилирование через `pprof`.

## 2. Этапы проектирования алгоритма

### 2.1. Формализация задачи

Запишите задачу в виде сигнатуры (как будущую функцию):

```go
// Find возвращает индекс target в отсортированном nums, иначе -1.
func Find(nums []int, target int) int
```

Уточните:

- **Предусловия** (preconditions): `nums` отсортирован по возрастанию; длина `n >= 0`.
- **Постусловия** (postconditions): если `target` есть — индекс `i` такой, что `nums[i] == target`; иначе `-1`.
- **Ограничения**: `n <= 10^6`, значения в `int32` и т.д.

### 2.2. Примеры и граничные случаи

| Случай | Вход | Ожидание |
|--------|------|----------|
| Пустой массив | `[]`, `5` | `-1` |
| Один элемент, попадание | `[3]`, `3` | `0` |
| Один элемент, промах | `[3]`, `5` | `-1` |
| Начало / конец | `[1,2,3]`, `1` / `3` | `0` / `2` |
| Дубликаты | `[1,2,2,2,3]`, `2` | любой валидный индекс (уточните в ТЗ) |

Граничные случаи закладывают в тесты **до** написания основной логики (TDD).

### 2.3. Выбор стратегии

| Подход | Когда применять | Пример |
|--------|-----------------|--------|
| Полный перебор | Маленький `n` | Проверка всех пар |
| Жадный | Локальный оптимум даёт глобальный | Интервалы, некоторые графы |
| Разделяй и властвуй | Задача делится на подзадачи | Merge sort, бинарный поиск |
| Динамическое программирование | Перекрывающиеся подзадачи + оптимальная подструктура | Рюкзак, LCS |
| Граф (BFS/DFS) | Связность, кратчайший путь в невзвешенном графе | Лабиринт |
| Структуры данных | Нужны быстрые операции | Куча, хеш-таблица |

### 2.4. Инвариант цикла

**Инвариант** — свойство, истинное до и после каждой итерации. Для бинарного поиска:

> Если `target` присутствует в `nums`, то его индекс лежит в отрезке `[left, right]` (включительно).

Доказательство корректности строится на сохранении инварианта и сходимости (`left > right` → элемент не найден).

### 2.5. Псевдокод → блок-схема → код

1. Псевдокод на естественном языке или в стиле CLRS.
2. Блок-схема (draw.io) — для ветвлений и циклов.
3. Реализация на Go с именами, отражающими инвариант (`left`, `right`, `mid`).

### 2.6. Верификация

- **Юнит-тесты** на примерах и границах.
- **Property-based** (опционально): `slices.IsSorted` + результат `Find` согласован с линейным поиском.
- **Бенчмарки** для сравнения с наивным решением на больших `n`.

## 3. Анализ задачи и инварианты

### 3.1. Математическая модель

Для поиска в отсортированном массиве:

- Множество индексов: `I = {0, 1, …, n-1}`.
- Отношение порядка: `nums[i] <= nums[j]` при `i < j`.
- Цель: найти `i*` такой, что `nums[i*] == target`, или доказать отсутствие.

### 3.2. Инвариант и монотонность

Обозначим `f(i) = nums[i] - target`. На отсортированном массиве последовательность `f(i)` неубывает (с учётом дубликатов — неубывает). Искомый индекс — точка, где `f(i) == 0`, или её нет.

Бинарный поиск использует **монотонный предикат** `P(i): nums[i] >= target` (для поиска левой границы) или классический вариант с тремя указателями.

### 3.3. Варианты бинарного поиска

| Вариант | Диапазон | Условие выхода | Типичное применение |
|---------|----------|----------------|---------------------|
| Закрытый `[left, right]` | оба включены | `left > right` | Классический поиск |
| Полуоткрытый `[left, right)` | `right` исключён | `left == right` | Lower bound |
| Поиск границы | `P(i)` монотонен на `0..n` | первый `i` с `P(i)==true` | Вставка, «первый >= x» |

В практической задаче ниже используется **закрытый интервал** `[left, right]`.

## 4. Выбор структур данных в Go

### 4.1. Слайс (`[]T`)

- Непрерывная память (для базового типа), индексация за **O(1)**.
- Длина `len`, ёмкость `cap`; append может переаллоцировать.
- Для алгоритмов «на месте» уточняйте, можно ли менять вход.

```go
nums := make([]int, 0, n) // предвыделение ёмкости
```

### 4.2. Карта (`map[K]V`)

- Среднее **O(1)** вставка/поиск; худший случай зависит от хеша.
- Порядок обхода не детерминирован (до Go 1.12+ randomization).

### 4.3. Куча (`container/heap`)

- `Push` / `Pop` за **O(log n)** — приоритетная очередь, Дейкстра, топ-K.

### 4.4. Стек и очередь

В Go часто используют слайс:

```go
stack := []int{}
stack = append(stack, x)
top := stack[len(stack)-1]
stack = stack[:len(stack)-1]
```

Для BFS — очередь через слайс с индексом головы или `container/list` (реже из-за аллокаций).

### 4.5. Указатели и значения

- Передавайте **большие структуры** по указателю.
- Для маленьких неизменяемых данных — по значению.
- Алгоритмы на слайсах обычно принимают `[]int`, не `*[]int`.

## 5. Псевдокод и блок-схемы

### 5.1. Псевдокод бинарного поиска

```
BINARY_SEARCH(nums, target):
    left  ← 0
    right ← length(nums) - 1

    while left <= right:
        mid ← left + (right - left) / 2
        if nums[mid] == target:
            return mid
        else if nums[mid] < target:
            left ← mid + 1
        else:
            right ← mid - 1

    return NOT_FOUND (-1)
```

**Почему `mid = left + (right-left)/2`**, а не `(left+right)/2`: избегаем переполнения в языках с фиксированной шириной `int` (в Go на 64-bit это реже проблема, но привычка полезна).

### 5.2. Элементы блок-схемы (draw.io)

| Фигура | Назначение |
|--------|------------|
| Овал | Начало / конец |
| Прямоугольник | Процесс (присваивание, вычисление) |
| Ромб | Условие (ветвление) |
| Параллелограмм | Ввод / вывод (опционально) |
| Стрелки | Поток управления |

## 6. Оценка сложности

### 6.1. Временная сложность

Обозначения: `n = len(nums)`, `T(n)` — число элементарных шагов.

- Каждая итерация бинарного поиска **делит интервал пополам**: длина `n → n/2 → n/4 → … → 1`.
- Число итераций ≤ `⌊log₂ n⌋ + 1`.
- **Время: O(log n)**; **память: O(1)** (итеративная версия).

Сравнение:

| Метод | Время | Память | Условие |
|-------|-------|--------|---------|
| Линейный поиск | O(n) | O(1) | любой массив |
| Бинарный поиск | O(log n) | O(1) | отсортированный |

### 6.2. Амортизированная и худшая сложность

Для одного вызова `Find` амортизация не нужна. Для структур с `append` учитывайте редкие **O(n)** переаллокации при росте слайса.

## 7. Паттерны реализации на Go

### 7.1. Идиоматичная сигнатура

```go
package binarysearch

// Find ищет target в отсортированном по возрастанию nums.
// Возвращает индекс или -1, если элемент отсутствует.
func Find(nums []int, target int) int
```

### 7.2. Итерация вместо рекурсии

Рекурсивный бинарный поиск проще для учебы, но даёт **O(log n)** глубины стека. В продакшене и на собеседованиях чаще пишут цикл.

### 7.3. Ранний выход и читаемость

```go
for left <= right {
    mid := left + (right-left)/2
    switch {
    case nums[mid] == target:
        return mid
    case nums[mid] < target:
        left = mid + 1
    default:
        right = mid - 1
    }
}
return -1
```

### 7.4. Обобщения (Go 1.18+)

Для любого упорядочиваемого типа:

```go
func FindOrdered[T constraints.Ordered](nums []T, target T) int
```

См. реализацию в `task/binary_search/find.go`.

### 7.5. Ошибки вместо «магических» -1

В API пакетов иногда возвращают `(int, bool)` или `error`:

```go
func Find(nums []int, target int) (index int, ok bool)
```

Для учебной задачи достаточно `-1`, но в библиотеках предпочтительнее явный `ok`.

### 7.6. Документация и примеры

```go
// ExampleFind демонстрирует успешный поиск.
func ExampleFind() {
    idx := Find([]int{1, 3, 5, 7}, 5)
    fmt.Println(idx)
    // Output: 2
}
```

Запуск: `go test -run Example`.

## 9. Типичные ошибки

| Ошибка | Следствие | Как избежать |
|--------|-----------|--------------|
| `right = mid` вместо `mid-1` при `nums[mid] > target` | Бесконечный цикл | Рисуйте интервал `[left,right]` |
| `left = mid` при `nums[mid] < target` | Застревание | Используйте `left = mid + 1` |
| `(left+right)/2` | Переполнение в других языках | `left + (right-left)/2` |
| Поиск в неотсортированном массиве | Неверный ответ | Проверка или сортировка O(n log n) |
| Путаница `left <= right` и `left < right` | Off-by-one | Зафиксируйте один шаблон |
| Игнор пустого слайса | Паника / неверный индекс | Отдельный тест `len==0` |

### 10.1. Условие

**Задача 1. Бинарный поиск в отсортированном массиве**

Дан целочисленный массив `nums`, отсортированный по неубыванию (допускаются повторения), и целое число `target`.

**Требуется:**

1. Спроектировать алгоритм: инвариант, псевдокод, оценка сложности.
2. Построить **блок-схему** в draw.io (файл в репозитории).
3. Реализовать на Go функцию `Find(nums []int, target int) int`.

5. (Дополнительно) Реализовать обобщённую версию `FindOrdered` для типа `constraints.Ordered`.

**Формат ответа:**

- Индекс `i`, где `nums[i] == target`, если элемент есть.
- `-1`, если элемента нет.

**Ограничения:** `0 <= len(nums) <= 10^6`, значения умещаются в `int`.

**Примеры:**

```

Выход: -1
```

Структура схемы:

4. Ветвление по `nums[mid]` относительно `target`.
5. Возврат `mid` или сужение интервала.
6. **Конец**.

### 10.5. Эталонное решение (кратко)

Инвариант: если `target` существует, его индекс ∈ `[left, right]`.

На каждом шаге сравниваем середину и отбрасываем половину интервала, сохраняя инвариант. При `left > right` интервал пуст — элемента нет.

### 10.6. Расширения (самостоятельно)

1. **Lower bound:** первый индекс `i`, где `nums[i] >= target`.
2. **Upper bound:** первый индекс `i`, где `nums[i] > target`.
3. **Количество вхождений** `target` через `upper - lower`.
4. **Поиск в повёрнутом отсортированном массиве** (LeetCode 33) — два бинарных поиска.
